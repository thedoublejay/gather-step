# v3.1 In-Progress Notes

This file tracks the indexing performance, storage, and security work completed
on `feat/v3.1-indexing-performance-storage-security` since the Phase 0 baseline
captured in [`v3.1-baseline.md`](./v3.1-baseline.md). It is **not** a release
announcement: the v3.1 release is gated on the production Oxc visitor / default
switch and the SWC removal, neither of which are complete. The crate version
deliberately stays at `3.0.0` until that gate clears.

## Highlights

- Parser backend boundary lets the indexer route between SWC and Oxc through one
  abstraction so the eventual switch is a one-line change instead of a fork.
- Oxc adapter scaffold (status, source-type, span mapping, import-binding
  extraction, top-level declared-name extraction) lands behind that boundary,
  with three parity tests pinning the behaviour against SWC across every
  TS/JS extraction fixture.
- Indexing throughput work avoids redundant cloning in the traversal source path,
  caches path-alias discovery, gates framework augmenters by language, and
  bypasses the index-size walk on the default fast path.
- Search index footprint drops several megabytes by replacing stored fields
  with fast-field decoding, compacting edge metadata tags, removing the
  redundant per-row `description` text in favour of a dedicated indexed-only
  `qualified_name` field, replacing the edge-kind index with counters,
  and storing 16-byte BLAKE3 prefixes for the per-file change-detection cache.
- Context-pack precompute is bounded by repo count, salts cache keys by
  compatibility, prunes stale entries on write, and preserves unrelated packs
  through reindexes.
- Analytics queue is bounded by repo count and now runs off the writer hot path
  so large workspaces no longer back-pressure storage commits with git history
  work.
- Workspace registry counts are refreshed from the final graph state at the
  end of an index run so cold/warm/partial reindexes report the same numbers
  the graph holds.
- Symlink-aware path guards reject symlinked deployment configs, ignore
  symlinked watch event paths, separate `git worktree add` arguments, and cap
  local config reads to bound parser memory on hostile inputs.
- Search query parser splits identifier separators (`-`, `_`, `.`, `/`) before
  parsing so snake-case and slash-bearing repo names tokenize the same way they
  index. Qualified impact queries fall back to the tail segment when the
  qualified form does not hit, restoring impact lookups against the new
  search compaction.

## Indexing

- Parser backend boundary in `gather-step-parser`: `TsJsParserBackend` decides
  between SWC and Oxc and `parse_ts_js_with_backend` is the single dispatch
  site that the production indexing pipeline calls.
- Oxc adapter scaffold lives in `ts_js_oxc.rs`. It exposes status mapping that
  matches SWC's `Parsed`/`Recovered`/`Unrecoverable` semantics, `SourceType`
  selection by extension (including JSX-aware `.cjs`/`.mjs`), span-to-source
  mapping, import-binding extraction, and top-level declared-name extraction
  (function, class, var, type alias, interface, enum, namespace, named default
  exports). The current production `TsJsParserBackend::current()` still returns
  SWC; flipping the default and removing SWC are explicitly the next step and
  are deferred behind the parity tests below.
- `oxc_parity` integration test guards three properties across every fixture in
  `tests/fixtures/extraction_fidelity/`:
    - Oxc parser status is never worse than SWC (parsed/recovered/unrecoverable
      ordering).
    - Oxc and SWC produce the same set of import / re-export bindings.
    - Oxc and SWC agree on the set of top-level declared identifier names.
- Path-alias discovery is cached during indexing so workspaces with many repos
  do not re-read tsconfig / package.json content for every file.
- Framework augmenters are gated by language so non-TS repos do not pay for
  TS-only augmentation traversals.
- Default index pipelines skip the size-only filesystem walk; size sampling
  is now opt-in.
- Traversal sources are no longer cloned per file when the indexer can borrow
  the original buffer.

## Performance

- Context-pack precompute is bounded by repo count and selects targets through
  a precomputed shortlist instead of scanning the full graph.
- Context-pack writes prune stale entries and preserve unrelated packs across
  reindexes.
- Cache keys for context packs are salted by compatibility so format-version
  changes invalidate without manual cleanup.
- Analytics queue depth is bounded by repo count.
- Git analytics now runs off the writer hot path; storage commits no longer
  block on history work.
- Search workflow:
    - Redundant `description` text field replaced by an indexed-only
      `qualified_name` field (lowercased, no stemming, no positions). Restores
      lowercase identifier matches (`createorder` -> `createOrder`) and
      repo-prefix queries (`service-a`) without bringing back the
      natural-language tokenizer chain. `SEARCH_INDEX_VERSION` is bumped to
      `1` so older on-disk indexes are rejected at open time.
    - Search hit decoding reads `is_exported` and `lang` from fast fields
      instead of stored fields.
    - Search flag decoding compacts the per-document overhead.
    - Deferred search commits are guarded by an RAII `DeferredCommitGuard`
      that rolls back unflushed writes on unwind.
- Storage compactions:
    - Edge metadata tags are stored with compact representations.
    - The `edges_by_kind` projection is replaced by counters.
    - File index state stores 16-byte BLAKE3 hash prefixes (collision-safe for
      this rebuildable cache) instead of full 32-byte digests. Incremental
      classification truncates snapshot hashes through
      `metadata::stored_content_hash` before comparing so the prefix store
      stays consistent across read paths.

## Storage

- `storage-report` CLI emits a structured footprint report with per-table and
  per-search-file byte breakdowns. The report path is now overridable through
  `--report-path` so CI runs can land artifacts in a known location.
- `storage-report` accepts `--storage` so the inspector can run against an
  arbitrary on-disk index, not just the workspace default.
- Generated schema baselines are reset for the v3.1 fresh-release stance:
  `METADATA_SCHEMA_VERSION = 0`, `SEARCH_INDEX_VERSION = 1` (bumped from `0`
  by the qualified-name field reintroduction), graph schema version `0`. None
  of these carry migration or upgrade branches; incompatible older state fails
  open at the underlying storage layer instead of running through custom
  upgrade UX.

## Security

- Watcher ignores symlinked event paths so symlink-cycles cannot drive
  unbounded reindex loops.
- `gather-step.local.yaml` and other local config reads are capped at a
  bounded byte budget so hostile configs cannot blow up parser memory.
- `git worktree add` arguments are passed positionally rather than glued into
  one shell string, removing a path-injection surface.
- Deployment topology config rejects symlinked paths so a hostile workspace
  cannot point the topology at out-of-tree YAML.

## Bug fixes

- Search queries split identifier separators (`-`, `_`, `.`, `/`) before
  parsing so doctor / search projection queries that previously missed
  snake-case symbols hit them again.
- Qualified impact queries fall back to the tail segment after the search
  compaction, restoring `Module::Symbol`-style lookups.
- Workspace registry counts are refreshed from the final graph at the end of
  an index run so the registry never drifts behind the graph after partial
  failures or self-healing reindexes.
- Incremental classification truncates new content hashes to the stored
  prefix length before comparing, so the 16-byte hash prefix store does not
  flag every previously-indexed file as modified.

## Test coverage

- Oxc parser parity tests across every TS/JS extraction fixture:
  status, import bindings, and top-level declared-name sets.
- Secret-surface MCP smoke test exercises the redaction surface end-to-end.
- Deployment-topology MCP tools test pins the public response shape.
- Benchmark harness samples resource peaks (max RSS, peak memory footprint,
  open FDs on Unix) so future v3.1 perf claims can be validated quantitatively.

## Deferred to v3.1 release

- Production Oxc visitor port. The current Oxc adapter handles status, source
  type, spans, imports, and top-level declarations. The full SWC visitor in
  `ts_js_swc.rs` covers framework-aware patterns (Nest decorators, Express
  routes, Mongoose schemas, etc.), JSX/HTML attribute conventions, type-only
  aliases for resolution, and many more shapes; that surface still has to be
  ported behind `TsJsParserBackend::Oxc` and gated by the parity tests above.
- Switching `TsJsParserBackend::current()` from `Swc` to `Oxc` in production.
- Removing the SWC backend, its dependency tree, and its 3.4k-line visitor.
- Bumping the Rust workspace version to `3.1.0` and shipping the matching
  website / changelog entries.

These notes will be promoted to `v3.1.0.md` when the deferred items land.
