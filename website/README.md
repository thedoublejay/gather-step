# Gather Step — website

Landing page + living documentation for Gather Step, built with
[Astro Starlight](https://starlight.astro.build/) and deployed to
[Cloudflare Pages](https://pages.cloudflare.com/).

## Structure

```
website/
├── astro.config.mjs           # Starlight config (title, sidebar, social, edit links)
├── src/
│   ├── content.config.ts      # docs collection wiring
│   ├── styles/custom.css      # design tokens — wire the external design here
│   └── content/docs/
│       ├── index.mdx          # landing page (splash template)
│       ├── guides/            # getting-started, installation, …
│       ├── concepts/          # conceptual docs (auto-sidebar)
│       ├── reference/         # API / CLI reference (auto-sidebar)
│       └── changelog.md       # will be generated by release-please later
└── public/favicon.svg         # placeholder favicon
```

## Local development

```bash
cd website
bun install --frozen-lockfile
bun run dev          # http://localhost:4321
bun run build        # static output → dist/
bun run preview      # serve built site locally
```

Use Bun as the package manager for this site and commit `bun.lock`.

## Deploying to Cloudflare Pages

This site is connected directly to Cloudflare Pages through Git integration.
Cloudflare builds on every push.

1. Push this repo to GitHub.
2. In the Cloudflare dashboard → **Workers & Pages** → **Create** → **Pages** →
   **Connect to Git** → pick the repo.
3. Build settings:
   - **Framework preset:** Astro
   - **Build command:** `bun run build`
   - **Build output directory:** `dist`
   - **Root directory:** `website`
   - **Build image/runtime:** Bun-enabled image or install Bun before build
4. Add your custom domain under the Pages project → **Custom domains**.

## Cost

- Cloudflare Pages — free tier: unlimited requests, unlimited bandwidth,
  500 builds/month. Well within range for a docs site.
- Domain: register through Cloudflare Registrar at-cost (~$10–15/year).

## Design

The visual design is being produced in a separate tool. When it lands, wire
the tokens into `src/styles/custom.css`. Starlight exposes `--sl-*` CSS
variables (colors, typography, spacing) that can be overridden without
forking components — see
[CSS & Tailwind](https://starlight.astro.build/guides/css-and-tailwind/).

If the design calls for Tailwind, add it via the
[Starlight Tailwind guide](https://starlight.astro.build/guides/css-and-tailwind/#tailwind-css)
rather than a bare `@astrojs/tailwind` install — Starlight ships a plugin
that preserves its theming.

## Living documentation — next steps

This scaffolding is the shell. To make the docs truly "living":

- [ ] Adopt [Conventional Commits](https://www.conventionalcommits.org/) in
      the indexer package.
- [ ] Wire [release-please](https://github.com/googleapis/release-please) to
      generate `changelog.md` automatically on each release.
- [ ] Add a docs build step that regenerates `reference/cli.md` and
      `reference/api.md` from source (e.g. via TypeDoc) so they can never
      drift from the code.
- [ ] Evaluate versioned docs before the next breaking documentation split —
      Starlight supports this via multiple content collections or via
      [`starlight-versions`](https://github.com/HiDeoo/starlight-versions).
