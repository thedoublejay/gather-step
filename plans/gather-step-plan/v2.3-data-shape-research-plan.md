# Gather Step Implementation Plan: Data-Shape Research Carry-Forward (v2.3)

**Status:** Implemented on active v2.3 branch: `feat/v2.3-data-shape-research`.
**Source:** Carries forward v2.2 data-shape research that was intentionally not implemented in v2.2.0.

## Summary

v2.2.0 shipped the scoped version of data-shape awareness: direct TypeScript field reader/writer edges, field-aware impact summaries, and conservative Mongoose migration sibling hints. This v2.3 plan collects the research that remained outside that release:

- TypeScript field access beyond direct dotted member expressions.
- Optionality-vs-filter mismatch inference.
- Runtime Mongo data-shape probing or probe-plan generation.
- Broader migration detection beyond same-file conservative Mongoose patterns.
- Broader end-to-end oracle coverage for the v2.2 response shape.

**ELI5:** v2.2 taught Gather Step to notice the obvious field and migration clues. v2.3 is about noticing the clues hidden behind local aliases, destructuring, migration variants, and runtime shape checks without pretending to be a full compiler or database scanner.

## Implementation Progress

Completed on `feat/v2.3-data-shape-research`:

- Phase A: typed direct field access now follows same-scope aliases and object destructuring aliases, and `projection_impact` labels direct vs local-alias evidence.
- Phase B: optional payload filter mismatch now surfaces as a standalone `projection_impact` risk hint and planning-pack unresolved gap when optional payload contracts and migration filters touch the same field.
- Phase C: migration sibling bands now include generated Mongo `$type` probe plans and add a planning gap when optional payload evidence requires runtime shape review.
- Phase D: Mongoose migration detection now supports imported local model definitions, additional static write methods, and multiple static collections in one migration file.
- Phase D: TypeORM migration detection now indexes static `queryRunner.query(...)` SQL table names and static `queryRunner` table-method targets as migration siblings.
- Phase E: response-shape coverage includes the migration probe-plan envelope, and a store-backed planning oracle covers optional payload contracts plus migration filters.

Still open:

- Automatic database probe execution remains intentionally out of scope.
- Additional non-Mongoose migration families beyond TypeORM remain future work.
- Dynamic collection/table names, TypeORM entity metadata, and SQL WHERE-field extraction remain conservative coverage gaps.

## Review Of v2.2 Research Not Implemented

### Field Access Research

- Alias-chain following was not implemented.
- Destructured rebind tracking was not implemented.
- Optional-chaining traversal beyond direct depth-1 access was not implemented.
- Dynamic property-key resolution was not implemented.
- Field edges through `any`, `unknown`, generic containers, and UI `Props` were documented as best-effort boundaries rather than implemented.

**ELI5:** v2.2 can read `item.workflow.stepIds` when `item` has a known type. It cannot yet prove that `const wf = item.workflow; wf.stepIds` means the same field.

### Data-Shape Risk Research

- Optionality-mismatch inference was not implemented.
- Runtime Mongo shape probing was not implemented.
- Gather Step does not yet generate or run `$type` / `$exists` aggregation checks for a field under review.

**ELI5:** v2.2 can remind a reviewer that a field is optional. It cannot yet say "this migration filter requires the field to exist, but the type says missing is valid" as a first-class risk finding.

### Migration Research

- Imported Mongoose model-to-collection resolution was not implemented.
- Dynamic collection names were not implemented.
- Multi-collection migrations were not implemented.
- TypeORM, Knex, Prisma, and Atlas migration coverage was not implemented.

**ELI5:** v2.2 recognizes a small set of Mongoose migration shapes. v2.3 can widen that recognition one proven pattern at a time.

### Verification Research

- The broad oracle-fixture response-shape path stayed as future end-to-end coverage.
- v2.2 used focused parser, analysis, CLI, MCP, and snapshot tests for the shipped slice.

**ELI5:** v2.2 got focused tests around the new behavior. v2.3 should add bigger scenario tests that prove the whole planning pack stays useful.

## Goals

- Convert v2.2 best-effort boundaries into explicit v2.3 workstreams with testable acceptance criteria.
- Keep each parser expansion conservative and explainable in output.
- Prefer generated probe plans before any command that connects to a database.
- Add migration support only when a fixture can prove the convention and a negative fixture can stop overreach.
- Preserve v2.2 behavior and compatibility while adding richer signals.

## Non-Goals

- Not a full TypeScript flow analyzer.
- Not arbitrary JavaScript runtime execution.
- Not automatic production database access.
- Not exhaustive migration-framework support in one release.
- Not a replacement for human review of migration scope.

## Phase A: Field Access Beyond Direct Dotted Reads

**ELI5:** Start with local, boring aliases that humans can see without running TypeScript. If the parser cannot explain the alias path in one screen, skip it and say why.

1. Add failing fixtures for same-scope alias reads and writes.
   - `const workflow = item.workflow; workflow.stepIds.push(id)`
   - `const wf = item.workflow; wf = nextWorkflow`
   - `→ verify: cargo test -p gather-step-parser field_access_alias -- --nocapture`
2. Add failing fixtures for object destructuring from a typed receiver.
   - `const { workflow } = item; workflow?.stepIds`
   - `const { workflow: wf } = item; wf.stepIds`
   - `→ verify: cargo test -p gather-step-parser field_access_destructuring -- --nocapture`
3. Implement a local alias table limited to same-function, same-file, statically typed roots.
   - No cross-function propagation.
   - No mutation-sensitive alias lifetime beyond the local scope.
   - No dynamic keys.
   - `→ verify: existing dynamic/computed skip tests still pass`
4. Extend analysis and CLI/MCP surfaces to label inferred alias evidence separately from direct evidence.
   - `→ verify: gather-step projection-impact shows direct vs local-alias evidence bands`

## Phase B: Optionality Mismatch Inference

**ELI5:** When a type says a field may be missing, and a migration/filter only touches records where it exists, Gather Step should raise a review note before code review misses the mismatch.

1. Add a failing analysis test for a type with `workflow?` and a migration filter requiring `{ workflow: { $type: 'object' } }`.
   - `→ verify: cargo test -p gather-step-analysis optionality_mismatch -- --nocapture`
2. Create a risk hint that joins `PayloadField.optional`, field access evidence, and migration filter literals.
   - `→ verify: risk appears only when optionality and filter evidence both exist`
3. Surface the hint in planning and change-impact packs.
   - `→ verify: cargo test -p gather-step-cli --test pack_oracle optionality_mismatch`
4. Document the confidence boundary.
   - `→ verify: docs state this is a static review signal, not proof of production data distribution`

## Phase C: Runtime Probe Plans

**ELI5:** Gather Step should first print the safe query a human can run. A later explicit opt-in can run it, but no v2.3 work should silently connect to a database.

1. Add a generated Mongo probe plan for an optional field under review.
   - Suggested probes: missing count, `$type` distribution, null count, object count.
   - `→ verify: CLI snapshot includes aggregation commands without credentials or sample values`
2. Add pack output that links the probe plan to the optionality mismatch hint.
   - `→ verify: planning pack includes the probe-plan reminder once, within the follow-up budget`
3. Decide whether execution belongs in v2.3 or a later release.
   - If implemented, require explicit connection configuration and redact all values from durable output.
   - `→ verify: no test fixture or snapshot stores secrets, URIs, or sampled documents`

## Phase D: Migration Detection Expansion

**ELI5:** Widen migration support by adding one recognizable convention at a time. Every positive fixture needs a negative fixture so Gather Step does not claim certainty where it only guessed.

1. Extend Mongoose model resolution beyond same-file declarations.
   - Resolve imported local model definitions when the collection name is statically available.
   - `→ verify: parser test detects imported model collection and rejects unresolved imports`
2. Add multi-collection migration representation.
   - Store multiple `MigratesCollection` edges when a migration clearly touches multiple static collections.
   - `→ verify: planning pack groups siblings by each collection without duplicating unrelated migrations`
3. Add one non-Mongoose migration family after fixture review.
   - Candidate order: TypeORM, Knex, Prisma, Atlas.
   - `→ verify: positive fixture, negative fixture, and documentation boundary land with the parser change`
4. Keep dynamic collection names as explicit coverage notes unless a safe static resolver exists.
   - `→ verify: dynamic collection fixture emits a conservative skip note, not a false edge`

## Phase E: End-To-End Oracles

**ELI5:** Unit tests prove each small part. End-to-end oracles prove the complete planning pack still tells the reviewer the right story.

1. Add a curated workspace fixture that combines optional fields, direct field access, alias field access, and migrations.
   - `→ verify: oracle fails before v2.3 implementation because expected v2.3 hints are absent`
2. Add response-shape snapshots for planning and change-impact packs.
   - `→ verify: snapshots preserve v2.2 fields and add v2.3 fields without breaking existing clients`
3. Add benchmark coverage only after fixture size is stable.
   - `→ verify: benchmark oracle remains deterministic across repeated runs`

## Sequencing

1. Phase A first if the goal is better code graph precision.
   - `→ verify: parser and projection-impact tests prove alias/destructuring coverage without false positives`
2. Phase B next if the goal is to attack the original migration-review failure mode.
   - `→ verify: pack output calls out optionality mismatch and recommends a probe plan`
3. Phase C after Phase B, unless the team decides probe generation is documentation-only.
   - `→ verify: generated probes are useful without requiring database access`
4. Phase D can run independently after fixture selection.
   - `→ verify: every new migration convention has positive and negative coverage`
5. Phase E closes the release once at least one implementation phase lands.
   - `→ verify: end-to-end pack oracle captures the shipped v2.3 behavior`

## Initial v2.3 Slice

Start with Phase B plus the Phase C generated probe plan. That path most directly addresses the original schema-vs-data drift incident, while Phase A and Phase D can land as independent follow-ups on the same branch or later PRs.
