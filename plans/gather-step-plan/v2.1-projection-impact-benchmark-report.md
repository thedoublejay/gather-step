## Findings

1. The dedicated projection-impact CLI is useful on the generic stale-projection scenario. It catches the exact class of issues that raw search misses: derived source field, missing backfill, missing index/search mapping, filter impact, and frontend-only focus risk.
2. ~~The planning-oracle benchmark harness does not yet support `target.kind = "field"`, so projection scenarios fail before scoring.~~ Follow-up review on the live PR worktree found this stale: `mode = "projection_impact"` now dispatches to the projection-impact command path. The later remaining gaps were scorer semantics: expected field-not-found projection scenarios were counted as empty-result regressions, and expected projection missing-evidence values were counted as generic unresolved gaps. Fixed on PR branch commit `ab66856`.
3. Task 1 and Task 3 link-quality gates failed because the benchmark scorer was stale, not because projection impact regressed. The scorer indexed the curated monorepo as one repo, only walked one outgoing hop, skipped virtual bridge nodes, missed inbound type consumers, and compared repo-relative graph paths against workspace-relative task paths. Task 3 also expected `Imports`, but the fixture intentionally emits `UsesTypeFrom` for `import type`. Fixed on PR branch commit `ca49911`.
4. ~~The existing `/Users/jjadonis/Documents/repos/gather-step-projection-impact` worktree has uncommitted edits that do not compile because `clap::value_parser!(usize).range(1..=100)` is unavailable with this lockfile.~~ Follow-up review on the live PR worktree found this stale: the CLI now uses a custom `parse_projection_impact_limit` parser and `cargo check -p gather-step --locked` passes. Extra parser coverage was still useful for `--limit 101` and non-integer values.
