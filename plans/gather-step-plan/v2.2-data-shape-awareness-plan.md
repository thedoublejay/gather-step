# Gather Step Implementation Plan: Data-Shape Awareness (v2.2)

**Status:** Active v2.2 follow-on after v2.1 projection impact released and completed on 2026-04-30. Workstream B was implemented on 2026-04-30; Workstream A remains planned.
**Motivation incident:** An April 2026 Mongo migration filter `{ workflow: { $type: 'object' } }` was inherited verbatim from a prior migration without re-evaluating the production data-shape distribution. The filter excluded a large set of records where `workflow` was missing entirely. Three review passes caught logic issues but none flagged that the filter scope did not match the field's actual shape distribution.

The bug class is **schema-↔-data drift**: source code (and types) admit a shape variant that the runtime data also exhibits, but a code site (filter, guard, narrow) excludes it. Gather Step today has partial signal that could surface the smell earlier; this plan makes that signal usable end-to-end and adds a domain-specific surface for Mongo migrations.

**ELI5:** Right now Gather Step knows TypeScript field optionality and which symbols touch a type, but it does not know which functions read or write a specific field, and it does not treat Mongo migrations as a special class of file. Two cheap additions raise the floor: (a) record direct member-access edges on typed receivers, (b) recognize migration files and surface prior migrations on the same collection when planning a new one.

## Goals

- Field-level reader/writer edges in the graph for direct member-access on a statically-resolvable receiver type.
- Migration-file recognition for mongoose-flavored repos, with a "prior migrations on this collection" surface injected into `planning` packs.
- Both features ship with explicit best-effort framing: misses on aliased reads (#4) and toolchains we do not detect (#5) must be documented in tool output, not papered over.
- Regression oracles that fail today against an optional `workflow` fixture and pass after the work lands.

## Non-Goals

- **Not a TypeScript flow analyzer.** No alias-chain following, no destructuring rebinds, no optional-chaining traversal beyond direct dotted access on the original binding, no dynamic property-key resolution. The full version of any of these competes with `tsserver` and is out of scope.
- **Not exhaustive migration-toolchain coverage.** Pick one or two conventions the user's stack actually uses (`mongoose-migrate` / `migrate-mongo` style: file exports `up`/`down`, mongoose driver imported, raw `db.collection(...)` calls or `Model.updateMany(...)`). Other toolchains (typeorm, knex, prisma, atlas) are explicit non-goals for v2.2 — capture them as backlog if a contributor wants to extend.
- **Not a runtime data probe.** Gather Step still does not connect to MongoDB. The runtime `$type` aggregation remains a separate human/runbook step. The point of #5 is to *remind* the operator to do that probe, not to replace it.
- **Not a v2.0 or v2.1 blocker.** v2.2 milestone, contributor-friendly, both workstreams independently shippable.

## Workstream A: Field-Level Reader/Writer Edges (Scoped)

**Scope:** record direct dotted member-access on a receiver whose type resolves through the existing symbol graph. `alert.workflow.taskIds` where `alert: Alert`. Skip everything else, document the skip.

**ELI5:** When you ask "what touches `Alert.workflow`?", today Gather Step answers "every symbol that touches `Alert`". After this workstream it answers "these specific functions read `workflow.taskIds`, these write it, and the type itself flags `workflow?` as optional — which means a filter that requires it to exist excludes a real branch."

### A0. Lock Review Contract And Failing Oracles

Implement tests before implementation changes.

1. Add `tests/fixtures/oracle/field_access_alert_workflow_optional/`.
   - Model an `Alert` interface with `workflow?: { taskIds: string[] }`.
   - Add a use case that does `alert.workflow.taskIds.push(id)` (writer).
   - Add a query that does `if (alert.workflow) return alert.workflow.taskIds` (reader).
   - Add a migration-style file with filter literal `{ workflow: { $type: 'object' } }`.
   - `→ verify: oracle fails before implementation because change_impact on Alert.workflow returns symbol-level results only, not field-granular reader/writer split, and does not flag the optionality-vs-filter mismatch.`

2. Add `tests/fixtures/oracle/field_access_aliased_read_negative/`.
   - Model `const { workflow } = alert; workflow?.taskIds`.
   - Assert that the aliased read is **not** captured (best-effort scope).
   - `→ verify: oracle fails if implementation accidentally tries to follow alias chains and produces a partial/incorrect alias edge — we want explicit "missed by design" coverage so future contributors do not mistake it for a bug.`

3. Extend oracle expectations.
   - Candidate fields: `expected_field_readers`, `expected_field_writers`, `expected_optionality_mismatches`, `forbidden_aliased_edges`.
   - → verify: `cargo test -p gather-step-analysis --test field_access -- --nocapture` reports clear assertion failures rather than ambiguous diffs.

### A1. Graph Vocabulary

1. Add `EdgeKind::FieldRead { field_path: Vec<String> }` and `EdgeKind::FieldWrite { field_path: Vec<String> }` (or string-encoded equivalents that match existing edge serialization conventions).
2. Add a graph-data version bump if the schema enum changes — clean-slate rebuild required, no compatibility shim.
   - → verify: `cargo test -p gather-step-storage` passes with rebuilt fixtures.

### A2. Parser Extraction (TypeScript First)

1. In the existing tree-sitter walk, recognize `member_expression` nodes whose receiver chain roots at a binding with a resolvable static type.
2. Resolve receiver type via the existing symbol-graph type binding (no new type inference).
3. For each direct dotted access, emit a `FieldRead` or `FieldWrite` edge from the enclosing symbol to the receiver type, carrying the field-path slice.
4. Skip with explicit no-op when the receiver:
   - is an alias (`const { x } = obj`, `const x = obj.x`)
   - resolves through optional chaining beyond depth 1
   - resolves to `any` / `unknown`
   - is a destructured rebind
5. Read vs. write disambiguation: assignment LHS, `Array.prototype` mutators on the chain tail, and `delete` are writes. Default to read.
   - → verify: `cargo test -p gather-step-parser` covers a positive case, an aliased-skip case, and a writer case.

### A3. Analysis And Surfaces

1. Extend `change_impact` so a query on `Alert` enumerates field-readers and field-writers as a separate output band, not mixed with symbol-level results.
2. When the receiver type's `PayloadField.optional == true` and a code site narrows by existence (`if (alert.workflow)`) but a *different* code site assumes presence (raw `alert.workflow.x`), surface that as an `optionality_mismatch` hint.
3. CLI: `gather-step impact Alert.workflow` (dotted symbol notation) returns the field-scoped slice.
4. MCP: `change_impact` tool gains an optional `field_path` parameter.
   - → verify: `cargo test -p gather-step-cli --test pack_oracle field_access` passes.

### A4. Documentation

1. Document explicitly in tool output and in `docs/`: "field-access edges are best-effort. Aliased reads, destructured rebinds, dynamic property keys, and reads through optional chaining beyond depth 1 are not tracked."
2. Add a section to the website docs under "What it does not do" so users do not over-trust the surface.

## Workstream B: Mongoose Migration Sibling Awareness

**Scope:** detect mongoose-flavored migration files, extract the target collection name from filter literals, and inject "prior migrations on this collection" + their filter literals into the `planning` pack when the anchor is a migration file. Mongoose only for v2.2.

**ELI5:** When you start writing a new migration on the `alert__alerts` collection, Gather Step opens the planning pack with a list of every prior migration that touched the same collection and the filter each one used. The new migration cargo-cults the prior filter? You see it immediately, before writing the code.

### B0. Lock Review Contract And Failing Oracles

**Implemented 2026-04-30:** Review contract is covered by focused parser/MCP unit tests plus the context-pack response-shape snapshot. The original oracle-fixture shape remains useful for broader integration coverage, but the v2.2 slice is guarded by targeted tests that exercise the same positive and negative cases without expanding the fixture workspace.

1. Add `tests/fixtures/oracle/migration_sibling_mongoose/`.
   - Two prior migration files on collection `alerts`, each with a different filter literal (`{ workflow: { $type: 'object' } }` and `{ workflow: { $exists: false } }`).
   - One new migration file on the same collection.
   - `→ verify: oracle fails before implementation because the planning pack on the new migration does not surface the prior two as siblings.`

2. Add `tests/fixtures/oracle/migration_sibling_unsupported_toolchain_negative/`.
   - A typeorm-style migration file.
   - Assert it is **not** detected and the planning pack does not pretend to.
   - `→ verify: oracle fails if implementation overreaches.`

3. Extend oracle expectations: `expected_migration_siblings`, `expected_collection_name`, `expected_prior_filters`.

### B1. Detection

**Implemented 2026-04-30:** `crates/gather-step-parser/src/frameworks/mongoose_migration.rs` detects conservative Mongoose migration files, extracts one `db.collection('<name>')` collection, captures `updateMany` filter literals, and rejects TypeORM-style or multi-collection migrations.

1. New module `crates/gather-step-parser/src/frameworks/mongoose_migration.rs`.
2. Detection heuristic (all must hold for a positive match):
   - File imports `mongoose` (existing `PackId::Mongoose` recognition).
   - File exports a `module.exports = { up, down }` or `export const up`/`export const down` or `export async function up`.
   - File path matches `**/migrations/**` or repo profile declares migration directory.
3. Extract collection name from `db.collection('<name>')` literal or `Model<Name>.updateMany(...)` where `Name` resolves to a model with a known collection.
4. Extract filter literal AST node for each top-level call. Store it serialized.
   - → verify: `cargo test -p gather-step-parser mongoose_migration` covers positive, negative-toolchain, and ambiguous-collection cases.

### B2. Graph Vocabulary

**Implemented 2026-04-30:** Mongoose augmentation emits `EdgeKind::MigratesCollection` from the migration `up` symbol to a virtual `__migration_collection__<name>` entity node. Filter literals are stored in tagged edge metadata for planning-pack display.

1. Virtual node convention: `__migration_collection__<name>` (mirrors existing `__topic__` pattern in `proofs.rs`).
2. Edge: `EdgeKind::MigratesCollection` from the migration file's primary symbol to the virtual collection node.
3. The virtual node clusters all prior migrations on that collection — sibling-lookup is a single graph query.
   - → verify: `cargo test -p gather-step-storage migration_collection_node` passes.

### B3. Pack Hint Injection

**Implemented 2026-04-30:** `context_pack` planning mode now returns a `migration_siblings` band for migration anchors. Entries include sibling file path, symbol id, optional latest indexed commit short SHA, and serialized filter literals. The verification hint is unconditional for detected migration anchors.

1. Extend `pack_assembly.rs` so a `planning` pack on a migration anchor includes a `migration_siblings` band.
2. Each sibling entry: file path, commit short SHA (via existing `gather-step-git` integration), serialized filter literal, optional one-line summary.
3. Inject a fixed hint string: **"Run `db.<collection>.aggregate([{ $group: { _id: { $type: '$<field>' }, count: { $sum: 1 } } }])` against a representative environment before trusting any filter that scopes by field type. Schema-↔-data drift is not detectable from source alone."**
4. The hint is unconditional on migration anchors. Cheap, high-signal, and survives users skimming the pack.
   - → verify: `cargo test -p gather-step-analysis --test pack_mode migration_planning_hint` passes.

### B4. CLI And MCP

**Implemented 2026-04-30:** MCP planning packs include the new response field. CLI JSON output receives it through the existing context-pack response path; no new command surface was added.

1. CLI: `gather-step pack <migration-file> --mode planning` returns the new band.
2. MCP: `context_pack` tool's planning mode auto-includes it when the anchor is a migration; no new tool surface needed.
   - → verify: `cargo test -p gather-step-mcp` passes.

### B5. Documentation

**Implemented 2026-04-30:** Added the data-shape verification guide and linked it from the context-packs docs/sidebar.

1. Document supported toolchain (mongoose only, v2.2). List unsupported toolchains explicitly so users know not to trust silence.
2. Add a "data-shape verification reminder" doc page under `website/src/content/docs/guides/` cross-linking to the planning hint.

## Cross-Cutting

- Schema bump: both workstreams add edge kinds. Coordinate the version bump so v2.2 is a single graph-rebuild bump, not two.
- Benchmark oracles: each fixture should add a benchmark-oracle entry if eligible, matching `projection-impact-plan.md` precedent.
- Repo profile (advisory only): a repo profile may surface its own migration directory convention. Keep the core detection generic; let the profile narrow the path glob.
- Braingent learning: the original incident is a Braingent-worthy learning record. That capture is independent of this public plan and should live outside `gather-step`.

## Open Questions

- **Field-path serialization format.** Vec<String> vs. dotted string. Pick one and pin it before A1 lands.
- **Edge cardinality budget.** A naive implementation could 5–10× graph size on TS-heavy repos. Measure on the bundled fixture corpus before merging A2; if it exceeds 2× we add a per-edge dedup/aggregation step.
- **Migration model resolution.** Does mongoose model → collection-name resolution require a second parser pass, or can it ride existing schema indexing? Prototype B1 step 3 first to find out.
- **Profile vs. core split.** Should migration directory globs live in core defaults (`**/migrations/**`) or only in repo profiles? Default to core with profile override.

## Sequencing

A and B are independent and can ship in either order. Recommended order: B first (smaller, higher signal-per-line, directly attacks the motivating incident), then A. Both behind the v2.2 milestone.
